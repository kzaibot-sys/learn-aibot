## ФАЗА 1 — КРИТИЧЕСКИЕ БАГИ (приоритет: максимальный)

### 1.1. Сессия не сохраняется (пользователя разлогинивает при обновлении страницы)

**Проблема:** После обновления страницы (F5) пользователь теряет авторизацию и его перекидывает на логин.

**Что сделать:**
- Проверить, как хранится токен авторизации (JWT, session cookie, localStorage).
- Если токен в `localStorage` — убедиться, что при инициализации приложения он читается и подставляется в заголовки API.
- Если используется JWT — реализовать refresh-token механизм:
  - Access token: короткоживущий (15-30 мин).
  - Refresh token: долгоживущий (7-30 дней), хранится в httpOnly cookie.
  - При истечении access token — автоматический запрос нового через refresh endpoint.
- Добавить persistent auth state: при загрузке приложения первым делом проверять наличие валидного токена.
- Если токен невалиден — тогда и только тогда редиректить на логин.

**Проверка:** Залогиниться → обновить страницу → пользователь остаётся залогиненным. Закрыть браузер → открыть снова → пользователь залогинен.

### 1.3. Локализация/переводы сломаны

**Проблема:** Некоторые элементы интерфейса показываются на казахском языке, хотя язык по умолчанию — русский. Кнопка смены языка не работает. Переводы в целом некорректные.

**Что сделать:**
- Язык по умолчанию: **русский (ru)**.
- Поддерживаемые языки: **русский (ru)**, **казахский (kk)**.
- Проверить ВСЕ файлы переводов. Убедиться, что:
  - У каждого ключа есть перевод на оба языка.
  - Нет пустых значений и `null`.
  - Нет ситуаций, где fallback-язык — казахский вместо русского.
- Установить `fallbackLng: 'ru'` в конфиге i18n.
- Починить кнопку смены языка в header:
  - При клике переключать язык ru ↔ kk.
  - Сохранять выбранный язык в `localStorage`.
  - При загрузке приложения читать сохранённый язык.
- Пройтись по ВСЕМ экранам и убедиться, что русский перевод полный и грамотный.
- Казахский перевод тоже проверить на корректность. Где перевод отсутствует или некорректен — написать корректный.

**Проверка:** При первом заходе — всё на русском. Переключить на казахский — всё переключилось. Обновить страницу — язык сохранился.

  --text-secondary: #6b7280;

  - Акцентный цвет остаётся одинаковый.
- Подключить кнопку переключения в header:

  - При загрузке — читать из `localStorage`, по умолчанию: `dark`.
  - Иконка кнопки: солнце ☀️ для переключения на светлый, луна 🌙 для тёмного.
- Переход между темами — плавная анимация (`transition: background-color 0.3s, color 0.3s`).

**Проверка:** Переключить тему → всё читаемо, красиво, нет артефактов. Обновить страницу — тема сохранена.

## ФАЗА 3 — СИСТЕМА УВЕДОМЛЕНИЙ

### 3.1. Реализовать систему уведомлений

**Что сделать:**
- **Бэкенд:**
  - API endpoint: `POST /api/notifications` — администратор создаёт уведомление (title, message, type).
  - API endpoint: `GET /api/notifications` — получить уведомления для текущего пользователя.
  - API endpoint: `PATCH /api/notifications/:id/read` — пометить как прочитанное.
  - Модель уведомления: `{ id, title, message, type, createdAt, isRead }`.
  - Типы: `info`, `warning`, `success`, `course_update`.

- **Фронтенд — кнопка уведомлений в header:**

  - Каждое уведомление: заголовок, текст, время, статус (прочитано/нет).
  - Клик по уведомлению — помечает прочитанным.
  - Кнопка «Прочитать все».

- **Звуковое уведомление:**
  - При получении нового уведомления воспроизводить короткий звук (notification sound).
  - Использовать `Audio API`. Звуковой файл — приятный короткий звук (создать или использовать свободный).
  - Polling: каждые 30 секунд проверять новые уведомления (или WebSocket, если позволяет стек).
  - Звук воспроизводить только если появилось новое уведомление, которого не было в предыдущем запросе.

- **Админ-панель — раздел «Уведомления»:**
  - Форма: заголовок, текст, тип уведомления.
  - Кнопка «Отправить всем».
  - История отправленных уведомлений.

**Проверка:** Админ отправляет уведомление → у пользователя появляется badge + звук → клик → открывается панель → уведомление помечается прочитанным.

## ФАЗА 4 — ВИДЕОПЛЕЕР

### 4.1. Собственный видеоплеер вместо YouTube embed

**Проблема:** Сейчас используется встроенный YouTube-плеер (iframe). Нужен свой кастомный плеер.

**Что сделать:**
- **Бэкенд — скачивание видео с YouTube:**
  - При добавлении курса администратор может вставить YouTube URL.
  - Сервер скачивает видео с YouTube (использовать `yt-dlp` — установить как зависимость).
  - Сохранять видео на сервере в папку `/uploads/videos/` или в S3-совместимое хранилище.
  - Сохранять метаданные: оригинальный URL, путь к файлу, размер, длительность.
  - Альтернативно: администратор может загрузить видеофайл напрямую (drag & drop или выбор файла).
  - Поддерживаемые форматы: mp4 (основной), webm.
  - При скачивании конвертировать в mp4 если формат другой (через `ffmpeg`).

- **Фронтенд — кастомный видеоплеер:**
  - Использовать HTML5 `<video>` тег как основу.
  - НЕ использовать YouTube iframe нигде.
  - Элементы управления (кастомные, не нативные):
    - Play / Pause (центральная кнопка + кнопка в панели).
    - Прогресс-бар с возможностью перемотки (drag + click).
    - Отображение текущего времени / общей длительности.
    - Регулировка громкости (слайдер + иконка с мутом).
    - Полноэкранный режим.
    - Скорость воспроизведения (0.5x, 0.75x, 1x, 1.25x, 1.5x, 2x).
    - Кнопка Picture-in-Picture (если поддерживается браузером).
    - Настройка качества (если видео хранится в нескольких разрешениях).
  - Дизайн:
    - Современный, минималистичный.
    - Полупрозрачная нижняя панель, которая скрывается при неактивности мыши (через 3 сек).
    - Плавные анимации появления/скрытия контролов.
    - Rounded corners на контейнере плеера.
    - Поддержка обеих тем (dark/light) через CSS-переменные.
  - Адаптивность:
    - На мобильных — увеличенные кнопки управления (touch-friendly, min 44px).
    - Свайп вверх/вниз для громкости (опционально).
    - Тап по центру — play/pause.
    - Двойной тап слева/справа — перемотка ±10 сек.
    - Ландшафтный режим — автоматический полный экран.
  - Прогресс просмотра:
    - Сохранять позицию просмотра для каждого видео (в БД или localStorage).
    - При повторном открытии — предложить продолжить с того места.
    - Отмечать видео как «просмотрено» при достижении 90% длительности.

**Проверка:** Видео загружается → плеер отображается → все контролы работают → на мобильном удобно → прогресс сохраняется.

## ФАЗА 5 — ПРОИЗВОДИТЕЛЬНОСТЬ И ОПТИМИЗАЦИЯ

### 5.1. Ускорение загрузки сайта и навигации

**Проблема:** Сайт загружается очень медленно. Переход между страницами (курсы и т.д.) тормозит. API-запросы медленные.

**Что сделать на фронтенде:**
- **Code splitting:** Каждая страница/route — отдельный чанк. Использовать `React.lazy()` + `Suspense` (или аналог для Vue/Next).
- **Lazy loading изображений:** Все `<img>` — с `loading="lazy"`. Использовать WebP формат.
- **Минимизация бандла:**
  - Проверить через `webpack-bundle-analyzer` (или аналог) что попадает в бандл.
  - Убрать неиспользуемые зависимости.
  - Tree shaking для библиотек.
- **Кэширование:**
  - API-ответы кэшировать на клиенте (React Query / SWR / аналог).
  - Стратегия `stale-while-revalidate`: показываем кэш, обновляем в фоне.
  - Список курсов, профиль, прогресс — кэшировать.
- **Skeleton loading:**
  - Вместо пустого экрана при загрузке — показывать skeleton-заглушки.
  - Для списка курсов, карточек, видеоплеера.
- **Prefetch:**
  - При наведении на ссылку — prefetch данных следующей страницы.
- **Оптимизация рендеринга:**
  - Мемоизация компонентов (`React.memo`, `useMemo`, `useCallback`).
  - Виртуализация длинных списков (если есть).

**Что сделать на бэкенде:**
- **Оптимизация API:**
  - Все запросы к БД — проверить на N+1 проблему.
  - Добавить индексы на часто используемые поля (user_id, course_id, status).
  - Использовать пагинацию где возможно.
  - Включить gzip/brotli сжатие ответов.
- **Кэширование на сервере:**
  - Кэшировать список курсов, категории, статичные данные (Redis или in-memory).
  - TTL: 5-15 минут для списков, 1 час для статичных данных.
- **Статические файлы:**
  - Отдавать через CDN или с правильными заголовками кэширования (`Cache-Control: max-age=31536000` для хешированных ассетов).
- **Оптимизация видео:**
  - Streaming видео через HTTP Range requests (не загружать целиком).
  - Видео отдавать из статической директории, не через API.

**Целевые метрики (Core Web Vitals):**
- First Contentful Paint (FCP): < 1.5 сек
- Largest Contentful Paint (LCP): < 2.5 сек
- Time to Interactive (TTI): < 3.5 сек
- Переход между страницами: < 300 мс (видимый контент)

**Проверка:** Открыть DevTools → Network → Throttle до "Fast 3G" → сайт должен загружаться за 3-4 сек. Навигация между страницами — мгновенная.
